Data Quality Report — Fintech Transaction Dataset

Report Type: Automated Pipeline Check · Post-Ingestion Review Reviewed by: Senior Data Quality Engineer

1. Executive Summary

The dataset is in poor overall health, with only 60% of records fully complete — well below the 95% completeness threshold expected for production-grade transaction data. Four out of ten records contain null values in critical fields (amount, status, merchant), which directly impacts downstream reconciliation, reporting, and fraud-detection workflows. No duplicates or invalid (negative/zero) amounts were detected, and one high-value transaction has been flagged for review.

2. Issues Found

Issue A — Missing amount Values (2 records)

Attribute Detail
Severity 🔴 HIGH
What it means Two transactions have no dollar amount recorded. These records cannot be reconciled, settled, or included in revenue reporting. They are effectively "ghost" transactions that will cause discrepancies between the ledger and the payment processor.
Likely cause Partial writes from the payment gateway — the authorization call may have succeeded but the response payload was dropped before the amount was persisted (e.g., a timeout, network interruption, or schema mismatch during ingestion). Alternatively, an upstream source system may be emitting incomplete event payloads.

Issue B — Missing status Value (1 record)

Attribute Detail
Severity 🔴 HIGH
What it means One transaction has no settlement status (e.g., pending, completed, declined). Without status, the transaction cannot be routed through the correct post-processing workflow — it will be invisible to the settlement engine and may result in funds being held or double-charged.
Likely cause Race condition or event-ordering issue in the ingestion pipeline — the status update event may have arrived before the base transaction was created, or the status field was conditionally omitted by the source API due to an unhandled edge case (e.g., a processor-specific response code that doesn't map to the internal enum).

Issue C — Missing merchant Value (1 record)

Attribute Detail
Severity 🟡 MEDIUM
What it means One transaction cannot be attributed to a merchant. This affects merchant-level analytics, interchange fee calculations, and category-based spend reporting. If this record belongs to a regulated merchant category (e.g., gambling, crypto), it also creates a compliance blind spot.
Likely cause The merchant descriptor may have been null or malformed in the acquirer's response, or the MID (Merchant ID) lookup in the enrichment layer failed silently without logging an error or applying a fallback value.

Issue D — High-Value Transaction Flagged ($22,000 at Walmart)

Attribute Detail
Severity 🟡 MEDIUM
What it means A single transaction of $22,000 at Walmart has been flagged as a statistical outlier. While not inherently invalid, this amount is atypical for a retail merchant and could indicate a data entry error (e.g., decimal-point shift: $220.00 → $22,000.00), a bulk/corporate purchase, or potential fraud. If legitimate, it may also trigger BSA/AML reporting thresholds (transactions at or above $10,000).
Likely cause Requires manual investigation. Most probable causes: (1) a decimal/currency-conversion error at the point of sale or gateway level, (2) a legitimate high-ticket purchase (e.g., electronics in bulk), or (3) a structuring/fraud attempt.
